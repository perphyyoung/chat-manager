# 文档标签功能设计

## Context

当前应用采用三栏布局：左侧文档列表、中间对话内容、右侧问题列表。数据层使用 SQLite 通过 Electron IPC 与渲染进程通信。状态管理使用 Pinia。

本次变更需要新增标签系统，支持全局标签管理和文档标签关联。采用**混合布局**方案：

- 左侧面板顶部放置标签筛选区
- 中间面板顶部放置当前文档的标签编辑区
- 文档列表项显示标签标识

## Goals / Non-Goals

**Goals:**

- 用户可以创建、删除全局标签
- 用户可以为文档添加/移除多个标签
- 用户可以通过标签筛选文档列表
- 标签展示清晰，不占用过多界面空间
- 标签数据持久化存储

**Non-Goals:**

- 标签颜色自定义（初期使用系统配色）
- 标签层级/嵌套
- 问题级别的标签
- 标签搜索/自动补全（初期版本）

## Decisions

### 1. 数据模型设计

**方案A: 独立 Tag 实体 + 关联表**

```
Tag { id, name, createdAt }
Document { ..., tags: Tag[] }
document_tags { documentId, tagId }
```

**方案B: 文档内嵌标签数组**

```
Document { ..., tagNames: string[] }
```

**选择: 方案A**

- 理由: 支持标签元数据扩展（如颜色、描述），便于标签管理和去重
- 权衡: 实现复杂度稍高，需要关联表

### 2. 标签筛选交互模式

**方案A: 单选模式** - 点击一个标签，显示该标签的文档
**方案B: 多选模式** - 可选择多个标签，显示满足任一标签的文档

**选择: 方案A（单选）+ "全部"选项**

- 理由: 简单直观，满足大部分使用场景
- 扩展: 后续可升级为多选

### 3. 标签在文档列表的展示

**方案A: 完整标签名** - 直接显示 "[工作] [重要]"
**方案B: 颜色圆点** - 用彩色圆点表示有标签
**方案C: 混合** - 显示前2个标签名 + 更多提示

**选择: 方案C（混合）**

- 理由: 平衡信息密度和可读性
- 实现: 最多显示2个标签，超出显示 "+N"

### 4. 标签编辑交互

**方案A: 行内编辑** - 直接在文档标题旁编辑
**方案B: 下拉面板** - 点击展开标签选择面板

**选择: 方案B（下拉面板）**

- 理由: 空间充足，可展示所有可用标签，支持新建标签

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 标签过多导致界面拥挤 | 标签列表支持滚动，文档项最多显示2个标签 |
| 标签数据与文档数据同步问题 | 统一通过 Pinia store 管理，确保数据一致性 |
| 删除标签时关联数据清理 | 使用外键级联删除或事务处理 |
| 性能问题（大量文档筛选） | 标签筛选在内存中进行，利用 Pinia 响应式特性 |

## Migration Plan

1. **数据库迁移**: 新增 tags 表和 document_tags 关联表
2. **代码迁移**: 按层实现（领域层 → 应用层 → 展示层）
3. **回滚策略**: 数据库迁移可逆，删除新增表即可

## Open Questions

- 标签名称是否允许重复？（建议不允许，大小写敏感）
- 删除标签时是否需要确认？（建议需要，影响多个文档）
